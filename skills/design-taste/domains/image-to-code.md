# Image-to-Code 领域 —— 图转码工作流

> 核心原则：**视觉网站任务，必须先出图再写代码。** 图是视觉真相的主要来源。

---

## 0. 何时触发

| 触发信号 | 说明 |
|---------|------|
| "帮我做一个 landing page" | 视觉任务 → 走图转码 |
| "设计一个官网首页" | 视觉任务 → 走图转码 |
| "把这张设计图转成代码" | 明确图转码 |
| "做一个产品展示页" | 视觉任务 → 走图转码 |
| "改一下这个按钮的颜色" | 代码修改 → 不走图转码 |
| "加一个响应式断点" | 代码修改 → 不走图转码 |

---

## 1. 强制工作流

```
Step 1: 生成设计图（生图工具）
  ↓
Step 2: 深度分析图片（视觉模型）
  ↓
Step 3: 实现前端代码
```

**⛔ 禁止跳过 Step 1 直接写代码。**
**⛔ 禁止跳过 Step 2 直接对着图写代码。**

### Step 1: 生成设计图

#### 1.1 图片数量规则（条件触发）

**默认：只生成 Hero section 设计图（1 张）。** 其余 section 从 Hero 图推断风格。

**触发新图的条件（满足任一即生成）：**

| 条件 | 说明 |
|------|------|
| 布局方向突变 | 从纵向变为横向滚动、从居中变为非对称 |
| 视觉密度突变 | 从 DENSITY 3 跳到 DENSITY 8（如从 Hero 到数据表） |
| 背景语境突变 | 从暗色变为亮色、从图片背景变为纯色 |
| 组件类型突变 | 从卡片变为表格、从图文变为纯数据 |

**成本估算：**

| 页面复杂度 | 典型图片数 | 预计耗时 |
|-----------|-----------|---------|
| 简单 landing（3 section） | 1-2 张 | 2-4 分钟 |
| 标准 landing（6 section） | 2-3 张 | 4-6 分钟 |
| 复杂页面（8+ section） | 3-4 张 | 6-8 分钟 |

**铁律：不生成重复风格的图。** 如果两个 section 布局/色彩/密度相同，用同一张图推断。

#### 1.2 图片质量要求

- 水平方向（16:9 或 16:10）
- 足够大，文字可读
- 每个 section 独立构图
- 不重复使用同一张图的不同裁剪

#### 1.3 生图提示词规则

⚠️ 生图提示词由 GLM-5.1 辅助撰写（见 `domains/image-generation.md`），不由本 domain 直接写。

### Step 2: 深度分析图片

用视觉模型分析每张生成的设计图：

#### 2.1 分析维度

| 维度 | 检查内容 |
|------|---------|
| **布局结构** | 几列？对齐方式？元素位置关系？ |
| **色彩方案** | 背景色、accent 色、文字色、渐变位置 |
| **排版层级** | 标题字号/字重、正文字号、行距、字间距 |
| **间距系统** | section padding、元素间距、留白比例 |
| **组件清单** | 按钮样式、卡片样式、图标风格、图片比例 |
| **动效暗示** | 哪些元素暗示应该有动效？hover/scroll/入场？ |
| **响应式线索** | 哪些布局在移动端需要退化？ |

#### 2.2 分析输出格式

```
## 设计图分析

### 布局
- 整体：{描述}
- 列数：{N}
- 对齐：{左/中/右}

### 色彩
- 背景：{hex}
- 文字：{hex}
- Accent：{hex}
- 特殊：{渐变/叠加}

### 排版
- H1：{字号/字重/行距}
- H2：{字号/字重/行距}
- Body：{字号/行距/颜色}

### 间距
- Section padding：{Y}
- 元素间距：{Y}

### 组件
- 按钮：{样式描述}
- 卡片：{样式描述}
- 图标：{库/风格}

### 动效
- {元素} → {动效类型}

### 响应式
- 移动端：{退化方案}
```

### Step 3: 实现前端代码

#### 3.1 实现原则

- **忠实于图** — 图是真相来源，不要"改进"成 AI 默认风格
- **像素级还原** — 颜色、字号、间距尽量匹配
- **合理推断** — 图里看不清的细节（hover 状态、响应式行为）按 §4 纪律补充
- **标注偏差** — 如果某处与图不同，标注原因

#### 3.2 常见陷阱

| 陷阱 | 表现 | 修复 |
|------|------|------|
| **图好看代码丑** | 生图精美但代码用了 Inter + slate-900 | 从图中提取实际字体/颜色 |
| **代码比图"安全"** | AI 自动把非对称改成居中 | 忠实于图的布局 |
| **动效退化** | 图暗示动效但代码是静态的 | 分析图的动效暗示并实现 |
| **响应式偷懒** | 移动端直接缩小而非重构 | 按 §3.F 退化规则处理 |
| **图片降级** | 图的精美图片被替换为纯色块 | 用 picsum seed 保持视觉质量 |

## 2. 图转码专用禁令

| # | 禁令 | 原因 |
|---|------|------|
| I01 | 一张大图包含所有 section | 文字太小无法分析，section 间无独立性 |
| I02 | 跳过分析直接写代码 | 视觉模型的分析是质量保证 |
| I03 | 分析后"改进"设计 | 忠实还原 > 自我发挥 |
| I04 | 卡片套卡片套卡片 | 图转码常见递归陷阱 |
| I05 | Hero 信息过载 | 第一屏放太多东西 |
| I06 | 微小 pill/label/tag 装饰 | AI 图转码默认装饰 |
| I07 | 假界面术语 | 图中的文字是设计元素，不是功能描述 |

## 3. 与生图 domain 的协作

本 domain 依赖 `domains/image-generation.md` 生成设计图。
生图时使用 image-generation 的生图提示词流程（GLM-5.1 辅助），生成后回到本 domain 进行分析和实现。

### 协作流程

```
image-to-code 触发
  ↓
确定 section 数量和内容
  ↓
调用 image-generation 为每个 section 生图
  ↓
回到 image-to-code 分析每张图
  ↓
实现前端代码
```

---

## 4. 质量自检

交付前确认：
- [ ] 每 section 有独立设计图
- [ ] 每张图已完成深度分析
- [ ] 代码忠实于图（颜色/字号/间距/布局）
- [ ] 无"改进"成 AI 默认风格
- [ ] 响应式退化已处理
- [ ] 动效暗示已实现
- [ ] 无卡片套卡片
- [ ] Hero 信息不过载
