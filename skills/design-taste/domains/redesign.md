# Redesign 领域 —— 旧项目改造协议

本领域覆盖**现有网站/App 的审计和升级**。不适用于 Greenfield 新建项目。

核心工作流三步走：**Scan（扫描）→ Diagnose（诊断）→ Fix（修复）**。
不重写，只改进。

---

## 1. 检测模式

误分类模式是重设计输出差的最大原因。第一步必须正确归类。

### 1.A 三种模式分类

| 模式 | 判断条件 | 处理方式 |
|------|---------|---------|
| **Greenfield** | 无现有站点，或全面翻新已获批 | 按 design-taste 主框架新建流程处理。旋钮基线从 Section 1 推导 |
| **Redesign - Preserve** | 现代化但不破坏品牌 | 先审计，提取品牌 Token，逐步演进。改视觉不改骨架 |
| **Redesign - Overhaul** | 在现有内容上建新视觉语言 | 视觉按 Greenfield 处理；保留内容和信息架构 |

**模棱两可时只问一次：**
> "这次重设计是保留现有品牌，还是视觉上从头开始？"

能从上下文推断则不提问，直接声明模式往下走。

### 1.B 审计先于改动

提出改动前，必须记录当前状态的完整快照：

- **品牌 Token** —— 主色/强调色、字体栈、Logo 处理方式、圆角体系
- **信息架构** —— 页面树结构、主导航层级、关键转化路径
- **内容块清单** —— 什么存在、什么有效、什么是填充物
- **保留模式** —— 签名交互、可识别的 Hero、文案语气
- **退役模式** —— AI 痕迹、破损布局、死链接、通用库存图片、性能陷阱
- **现有站点旋钮读数** —— 推断当前 DESIGN_VARIANCE / MOTION_INTENSITY / VISUAL_DENSITY。这是起点，不是基线
- **SEO 基线** —— 当前排名页、meta 标题、结构化数据、OG 卡片

> ⚠️ **SEO 迁移是重设计 #1 风险。** 改 URL 结构可能导致排名清零。

### 1.C 保留规则

| 规则 | 说明 |
|------|------|
| **不改信息架构** | 除非被要求。保持页面 slug、锚点 ID、主导航标签稳定（SEO + 肌肉记忆） |
| **提取品牌色后再应用色彩规则** | 已经是紫色的品牌保持紫色——应用突破条件而非硬禁止 |
| **保留文案语气** | 除非要求重写。视觉现代化 ≠ 内容重写 |
| **尊重现有无障碍成果** | 不退化 focus state、alt text、键盘导航、对比度 |
| **尊重现有分析事件** | 不改按钮名、表单字段名、section ID（会破坏追踪数据） |

### 1.D 现代化杠杆

按顺序应用——Brief 满足即停：

| # | 杠杆 | 说明 |
|---|------|------|
| 1 | **字体刷新** | 每单位风险的最大视觉提升 |
| 2 | **间距与节奏** | 增加 section padding，修复垂直节奏 |
| 3 | **色彩校准** | 降低饱和度、统一中性色、保留品牌强调色 |
| 4 | **动效层** | 添加合适 MOTION_INTENSITY 的微交互到现有组件 |
| 5 | **Hero 和关键 section 重组** | 用参考模式词汇重构顶部漏斗 |
| 6 | **完整块替换** | 仅当现有块不可救药时 |

### 1.E 决策树：定向演进 vs 全面重设计

| 条件 | 决策 | 价值/风险比 |
|------|------|------------|
| IA、内容、SEO 健全 | **定向演进**（杠杆 1-4） | ~70% 价值在 ~40% 风险 |
| 视觉债务是结构性的（IA 破损、无设计系统、移动端破损） | **全面重设计**，严格保留内容 | 高风险高回报 |
| 品牌本身在变化 | **Greenfield** | 按新建流程处理 |

### 1.F 永不静默更改的内容

以下内容未经用户明确批准，**永不修改**：

- URL 结构 / 路由 slug
- 主导航标签
- 表单字段名或顺序（破坏分析追踪 + 浏览器自动填充）
- 品牌 Logo 或文字标记
- 现有法律/同意/Cookie 文案

## 2. 审计清单

逐项扫描现有项目，记录每个问题。这是诊断阶段的核心产出。

### 2.A Typography 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| T1 | **浏览器默认字体或全站 Inter** | `font-family` 只有系统字体或只有 Inter | 替换为有特色的字体：`Geist`、`Outfit`、`Cabinet Grotesk`、`Satoshi`。编辑/创意项目用衬线标题 + 无衬线正文配对 |
| T2 | **标题缺乏存在感** | 标题和正文视觉权重差距不够 | 增大显示字号、收紧 letter-spacing、减小 line-height。标题应感觉厚重且有意图 |
| T3 | **正文过宽** | 段落无 `max-width` 或宽度过大 | 限制段落宽度约 65 字符（`max-width: 65ch`）。增大 line-height 提升可读性 |
| T4 | **只用 Regular (400) 和 Bold (700)** | `font-weight` 只有两个值 | 引入 Medium (500) 和 SemiBold (600) 实现更细腻的层级 |
| T5 | **数字用比例字体** | 数据列不对齐、宽度跳动 | 数据密集界面用等宽字体或启用 `font-variant-numeric: tabular-nums` |
| T6 | **缺少 letter-spacing 调整** | 标题和标签的 tracking 相同 | 大标题用负 tracking（`letter-spacing: -0.02em`），小写/标签用正 tracking |
| T7 | **全大写副标题泛滥** | 多处 `text-transform: uppercase` | 尝试小写斜体、首字母大写或小型大写替代 |
| T8 | **孤立词** | 标题最后一行只剩一个词 | 用 `text-wrap: balance` 或 `text-wrap: pretty` 修复 |

### 2.B Color & Surfaces 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| C1 | **纯 `#000000` 背景** | `background: #000` 或 `#000000` | 替换为 off-black、深炭色或深色调（`#0a0a0a`、`#121212` 或深海军蓝） |
| C2 | **过饱和强调色** | accent 色 HSL 饱和度 > 80% | 饱和度保持在 80% 以下。降低饱和度使其与中性色融合，而非尖叫 |
| C3 | **多个强调色** | 出现 > 1 个 accent 色 | 只选一个。移除其余。一致性胜过多样性 |
| C4 | **冷暖灰混用** | Zinc/Slate/Stone 混用，或暖灰 + 冷灰交替 | 坚持一种灰色系。所有灰色用一致色调（暖或冷，不能两者兼有） |
| C5 | **AI 紫/蓝渐变审美** | 渐变从紫到蓝或蓝到青，Hero 区大量使用 | 这是最常见的 AI 设计指纹。替换为中性基色 + 单一精心挑选的强调色 |
| C6 | **通用 `box-shadow`** | 纯黑阴影 `rgba(0,0,0,0.x)` | 阴影调色匹配背景色。用有色阴影替代纯黑低透明度 |
| C7 | **纯平面无纹理** | 背景纯色无任何纹理 | 添加细微噪点、颗粒或微图案。纯平面矢量感觉冰冷 |
| C8 | **均匀渐变** | 线性 45° 渐变无变化 | 用径向渐变、噪点覆盖或网格渐变打破均匀性 |
| C9 | **光照方向不一致** | 不同 section 阴影方向各异 | 审计所有阴影，确保它们暗示单一一致的光源方向 |
| C10 | **暗色突兀段落** | 浅色页面中间突然出现深色 section | 要么做完整暗色模式，要么保持全页一致背景色调。需要对比时用同一色板稍深的色调——不要在米色页面中间突然跳到 `#111` |
| C11 | **空洞平面节** | 只有文字在纯色背景上的 section | 添加高质量背景图片（模糊/叠加/遮罩）、微图案或环境渐变。用 `picsum.photos/seed/{name}/1920/1080` 作为可靠占位来源 |

### 2.C Layout 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| L1 | **全居中对称** | 所有 section 内容居中、等宽 | 用偏移边距、混合宽高比或左对齐标题打破对称 |
| L2 | **三等列卡片作为 feature 行** | `grid-cols-3` 或 3 个等宽 flex 子项 | 最通用的 AI 布局。替换为 2 列交错、非对称网格、水平滚动或 masonry |
| L3 | **用 `height: 100vh` 做全屏** | CSS 中出现 `height: 100vh` | 替换为 `min-height: 100dvh` 防止移动端浏览器（iOS Safari 视口 bug）布局跳动 |
| L4 | **复杂 flexbox 百分比数学** | `w-[calc(33%-1rem)]` 等手工计算 | 用 CSS Grid 替代可靠的多列结构 |
| L5 | **无 max-width 容器** | 内容在宽屏上边缘到边缘拉伸 | 添加容器约束（约 1200-1440px）加 auto margins |
| L6 | **flexbox 强制等高卡片** | `items-stretch` 在内容长度不同时 | 允许可变高度或用 masonry |
| L7 | **统一 border-radius** | 所有元素用同一个 `rounded-xl` | 变化圆角：内部元素更紧（`rounded-lg`），容器更柔和（`rounded-2xl`） |
| L8 | **无重叠或深度** | 所有元素平铺相邻，无层叠 | 用负边距或 `z-index` 创造分层和视觉深度 |
| L9 | **对称垂直 padding** | `py-16` 上下永远相同 | 光学调整——底部 padding 通常需要稍大 |
| L10 | **Dashboard 总是左侧栏** | 布局固定为左侧 sidebar + 右侧内容 | 尝试顶部导航、浮动命令菜单或可折叠面板 |
| L11 | **缺少留白** | 元素紧密排列无呼吸空间 | 间距翻倍。让设计呼吸。密集布局适用于数据仪表盘，不适用于营销页 |
| L12 | **卡片组中按钮未底部对齐** | 不同内容长度的卡片 CTA 高度不一致 | 将按钮 `mt-auto` 固定在每张卡片底部，形成干净的水平线 |
| L13 | **feature 列表起始位置不一致** | 定价表/比较卡片中 feature 列表 Y 位置不同 | 用固定高度的标题/价格块或列表上方的一致间距，确保所有列对齐 |
| L14 | **并排元素垂直节奏不一致** | 卡片/列/面板间标题、描述、价格、按钮基线未对齐 | 对齐所有项中的共享元素。未对齐的基线使布局看起来破碎 |
| L15 | **数学对齐但光学不对** | 图标旁文字、圆形中的播放按钮偏移 | 数学居中不一定看起来居中。通常需要 1-2px 光学调整 |

### 2.D Interactivity 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| I1 | **按钮无 hover 状态** | 按钮 CSS 无 `:hover` 定义 | 添加背景偏移、轻微缩放或 hover 时 translate |
| I2 | **无 active/pressed 反馈** | 按钮 CSS 无 `:active` 定义 | 按下时添加微妙 `scale(0.98)` 或 `translateY(1px)` 模拟物理点击 |
| I3 | **即时过渡零时长** | 无 `transition` 或 `transition-duration: 0` | 所有交互元素添加平滑过渡（200-300ms） |
| I4 | **缺少 focus ring** | 无 `:focus` 或 `:focus-visible` 样式 | 确保键盘导航可见焦点指示器。这是无障碍要求，不可选 |
| I5 | **无 loading 状态** | 异步操作无加载指示器 | 通用圆形 spinner 替换为匹配布局形状的骨架屏 |
| I6 | **无 empty 状态** | 空列表/仪表盘空白无内容 | 设计有构图的"开始使用"视图 |
| I7 | **无 error 状态** | 表单错误用 `alert()` 或无提示 | 添加清晰内联错误消息 |
| I8 | **死链接** | 按钮链接到 `#` | 要么链接到真实目标，要么视觉禁用（`pointer-events-none` + 降低不透明度） |
| I9 | **导航中无当前页指示** | 所有 nav 链接样式相同 | 活动导航链接用不同样式（下划线、加粗、背景色等） |
| I10 | **锚点跳转** | 锚点点击瞬间跳转 | 添加 `scroll-behavior: smooth` |
| I11 | **用 `top`/`left`/`width`/`height` 做动画** | CSS transition 属性包含布局属性 | 切换到 `transform` 和 `opacity` 实现 GPU 加速平滑动画 |

### 2.E Content 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| W1 | **通用名如 "John Doe"** | 用户名/推荐人是常见假名 | 用多样化的真实感名字（多文化、多性别） |
| W2 | **假整数如 `99.99%`、`$100.00`** | 数据全是完美圆数 | 用有机不规整数据：`47.2%`、`$99.00`、`+1 (312) 847-1928` |
| W3 | **占位公司名如 "Acme Corp"** | 客户/合作伙伴名是明显假名 | 创造符合上下文的可信品牌名 |
| W4 | **AI 文案陈词** | 出现 "Elevate"、"Seamless"、"Unleash"、"Next-Gen"、"Game-changer"、"Delve"、"Tapestry"、"In the world of..." | 写平实具体的语言。每个动词指向真实功能 |
| W5 | **成功消息中的感叹号** | "操作成功！"、"已完成！" | 去掉感叹号。自信但不吵 |
| W6 | **"Oops!" 错误消息** | "出错了！"、"Oops! Something went wrong" | 直接陈述问题："连接失败，请重试" |
| W7 | **被动语态** | "修改已保存"、"错误已被记录" | 用主动语态："保存成功"、"我们未能保存你的修改" |
| W8 | **所有博客文章日期相同** | 多篇文章日期全是同一天 | 随机化日期使其看起来真实 |
| W9 | **多个用户同一头像** | 不同人用同一张头像图片 | 每个不同的人用独特素材 |
| W10 | **Lorem Ipsum** | 出现拉丁文占位文本 | 永远不用占位拉丁文。写真实草稿文案 |
| W11 | **全大写标题** | 标题全部大写如 "CONTACT US" | 用首字母大写（sentence case）替代 |

### 2.F Component Patterns 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| P1 | **通用卡片外观** | 卡片全是 border + shadow + 白色背景 | 去掉边框，或只用背景色，或只用间距。卡片只在 elevation 传达层级时才存在 |
| P2 | **总是一个 filled + 一个 ghost 按钮** | 每组按钮总是这个组合 | 添加文字链接或三级样式减少视觉噪音 |
| P3 | **Pill 形 "New" 和 "Beta" 徽章** | `rounded-full` 小标签 | 尝试方形徽章、旗帜或纯文字标签 |
| P4 | **手风琴 FAQ section** | 点击展开/折叠的标准 FAQ | 用并排列表、可搜索帮助或内联渐进式展示 |
| P5 | **3 列评价轮播 + 圆点** | testimonials 三张卡片 + 底部圆点指示器 | 替换为 masonry 墙、嵌入社交帖子或单条旋转引用 |
| P6 | **3 列定价表** | 定价三个等宽列 | 用颜色和强调突出推荐层级，不只是额外高度 |
| P7 | **到处用 modal** | 简单操作也弹出弹窗 | 简单操作用内联编辑、滑出面板或可展开 section 替代弹窗 |
| P8 | **圆形头像** | 头像全是 `rounded-full` 圆形 | 尝试 squircles 或圆角方形，减少通用感 |
| P9 | **日/夜切换总是太阳/月亮** | 主题切换是标准太阳月亮图标 | 用下拉菜单、系统偏好检测或集成到设置中 |
| P10 | **Footer 链接农场 4 列** | footer 四列等宽链接 | 简化。聚焦主要导航路径和法律要求的链接 |

### 2.G Iconography 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| G1 | **只用 Lucide 或 Feather 图标** | 图标库只装了 Lucide/Feather | 这些是 AI 默认图标选择。用 Phosphor、Heroicons 或自定义集区分 |
| G2 | **老套隐喻** | 火箭代表 "Launch"，盾牌代表 "Security" | 用不太陈旧的隐喻（bolt、fingerprint、spark、vault） |
| G3 | **stroke 宽度不一致** | 不同图标粗细各异 | 审计所有图标，统一为一种 stroke 重量 |
| G4 | **缺少 favicon** | `<head>` 中无 favicon 链接 | 始终包含品牌 favicon |
| G5 | **通用 "多元团队" 照片** | About 页用不自然库存团队照 | 用真实团队照片、抓拍或一致的插画风格替代 |

### 2.H Code Quality 审计

| # | 问题 | 检测方法 | 修复方案 |
|---|------|---------|---------|
| Q1 | **Div 汤** | 大量无语义 `<div>` 嵌套 | 用语义 HTML：`<nav>`、`<main>`、`<article>`、`<aside>`、`<section>` |
| Q2 | **Inline style 混用 CSS class** | `style="..."` 和 className 交替 | 所有样式移到项目的样式系统 |
| Q3 | **硬编码像素宽度** | `width: 800px` 等固定值 | 用相对单位（`%`、`rem`、`em`、`max-width`）实现灵活布局 |
| Q4 | **缺少 alt text** | `<img>` 无 alt 或 `alt="image"` | 为屏幕阅读器描述图片内容。有意义图片永远不要留空 |
| Q5 | **乱用 z-index** | `z-index: 9999` 等随意大值 | 在主题/变量中建立干净的 z-index 层级（1-10 足够） |
| Q6 | **死代码** | 大段注释掉的代码残留 | 交付前移除所有调试残留 |
| Q7 | **Import 幻觉** | import 了不存在的模块 | 检查每个 import 是否实际存在于 `package.json` 或项目依赖中 |
| Q8 | **缺少 meta tags** | 无 `<title>`、`description`、`og:image` | 添加完整的 SEO 和社交分享 meta tags |

### 2.I Strategic Omissions 审计（AI 通常忘记的）

| # | 遗漏 | 影响 | 修复方案 |
|---|------|------|---------|
| S1 | **无法律链接** | 合规风险 | 在 footer 中添加隐私政策和服务条款链接 |
| S2 | **无返回导航** | 用户流程死胡同 | 每个页面都需要返回方式（面包屑、返回按钮、logo 链接首页） |
| S3 | **无自定义 404 页面** | 用户体验断裂 | 设计有帮助的、品牌化的"页面未找到"体验 |
| S4 | **无表单验证** | 提交无效数据 | 添加邮箱、必填字段和格式检查的客户端验证 |
| S5 | **无 skip-to-content 链接** | 键盘用户无法快速到达内容 | 添加隐藏的 skip-link（`sr-only`，focus 时可见） |
| S6 | **无 cookie consent** | 法律合规风险 | 如果司法管辖区要求，添加合规的同意横幅 |

## 3. 升级技术

审计完成后，从以下高影响力技术中选择合适手段替换通用模式。
根据项目旋钮值（VARIANCE / MOTION / DENSITY）选择合适的技术——不是每项都用。

### 3.1 字体升级

| 技术 | 效果 | 最低旋钮要求 | 实现方式 |
|------|------|------------|---------|
| **可变字体动画** | 滚动或 hover 时插值字重或宽度，文字有生命力 | MOTION ≥ 4 | CSS `font-variation-settings` + JS 滚动插值，或 Motion `useTransform`
| **轮廓到填充过渡** | 文字从描边轮廓开始，在滚动进入或交互时填充颜色 | MOTION ≥ 3 | `-webkit-text-stroke` + `color: transparent` → 填充色 transition
| **文字遮罩揭示** | 大号排版作为窗口展示背后的视频或动画 | MOTION ≥ 5 | `background-clip: text` + 背景视频/渐变，滚动控制裁切区域 |

### 3.2 布局升级

| 技术 | 效果 | 最低旋钮要求 | 适用场景 |
|------|------|------------|---------|
| **破碎网格 / 非对称** | 元素故意忽略列结构——重叠、出血到屏幕外或带计算随机性的偏移 | VARIANCE ≥ 6 | 创意展示页、作品集、品牌故事 |
| **留白最大化** | 激进使用负空间将焦点强制到单一元素 | DENSITY ≤ 3 | 高端消费品牌、奢侈品、极简产品页 |
| **视差卡片堆叠** | 在滚动过程中固定并物理堆叠的 section | MOTION ≥ 5 | 功能展示、步骤流程、数据叙事 |
| **分屏滚动** | 屏幕两半向相反方向滑动 | MOTION ≥ 7, VARIANCE ≥ 6 | 对比叙事、前后对比、双产品展示 |

**移动端覆盖：** 所有非对称布局在 < 768px 必须退化为单列。不是可选。

### 3.3 动效升级

| 技术 | 效果 | 最低旋钮要求 | 注意事项 |
|------|------|------------|---------|
| **带惯性的平滑滚动** | 将滚动与浏览器默认解耦，获得更重、电影般的感觉 | MOTION ≥ 6 | 必须尊重 `prefers-reduced-motion`，提供即时降级 |
| **交错入场** | 元素以轻微延迟级联进入，结合 Y 轴平移和透明度淡入 | MOTION ≥ 3 | 永远不要一次性挂载所有内容。延迟间隔 60-100ms |
| **弹簧物理** | 用基于弹簧的运动替代线性缓动，获得自然有重量的感觉 | MOTION ≥ 3 | 推荐 `cubic-bezier(0.32, 0.72, 0, 1)` 或 spring 参数 `stiffness: 100, damping: 20` |
| **滚动驱动揭示** | 内容通过扩展遮罩、擦除或 SVG 路径绘制进入 | MOTION ≥ 5 | 用 `IntersectionObserver` 或 CSS scroll-driven animations，禁止 `window.addEventListener('scroll')` |

### 3.4 表面升级

| 技术 | 效果 | 实现要点 |
|------|------|---------|
| **真正的玻璃态** | 超越 `backdrop-filter: blur`。添加 1px 内边框和微妙内阴影模拟边缘折射 | `border: 1px solid rgba(255,255,255,0.15)` + `box-shadow: inset 0 1px 0 rgba(255,255,255,0.2)` |
| **聚光灯边框** | 卡片边框在光标下动态发光 | JS 追踪光标位置 + CSS `radial-gradient` 在边框上 |
| **颗粒和噪点覆盖** | 固定、`pointer-events: none` 的覆盖层，用微妙噪点打破数字平面感 | `position: fixed; pointer-events: none; opacity: 0.03-0.05;` + SVG noise filter |
| **有色调色阴影** | 携带背景色调而非通用黑色的阴影 | 蓝 → 蓝阴影、绿 → 绿阴影。`rgba(bg_r, bg_g, bg_b, 0.15)` |

## 4. 修复优先级

按此顺序应用变更，以获得最大视觉影响和最小风险：

| 步骤 | 操作 | 风险等级 | 预期收益 |
|------|------|---------|---------|
| **1** | **字体替换** —— 最大的即时改进，最低的风险 | 🟢 低 | ⭐⭐⭐⭐⭐ 极高 |
| **2** | **色板清理** —— 移除冲突或过饱和的颜色，统一中性色 | 🟢 低 | ⭐⭐⭐⭐ 高 |
| **3** | **Hover 和 active 状态** —— 让界面感觉有生命力 | 🟢 低 | ⭐⭐⭐⭐ 高 |
| **4** | **布局和间距** —— 正确网格、max-width、一致 padding | 🟡 中 | ⭐⭐⭐⭐ 高 |
| **5** | **替换通用组件** —— 用现代替代方案替换陈词模式 | 🟡 中 | ⭐⭐⭐ 中 |
| **6** | **添加 loading、empty、error 状态** —— 让界面感觉完成 | 🟢 低 | ⭐⭐⭐ 中 |
| **7** | **微调排版层级和间距** —— 高级的最终润色 | 🟢 低 | ⭐⭐⭐ 中 |

### 每步详细说明

**步骤 1 - 字体替换：**
- 检查当前 `font-family` 声明
- 选择替代字体（参考 design-taste 主框架 Section 4.1 字体规则）
- 替换字体栈，保留 fallback 链
- 验证所有字号在正文/标题/导航中仍然可读
- 风险：极低，纯 CSS 变更

**步骤 2 - 色板清理：**
- 审计所有颜色值，识别 accent 色数量
- 选择一个主 accent 色，移除其余
- 统一中性色为一种灰色系
- 检查对比度符合 WCAG AA
- 风险：低，纯 CSS 变更

**步骤 3 - 交互状态：**
- 为所有 `<button>` 和 `<a>` 添加 `:hover` 状态
- 添加 `:active` 反馈（`scale(0.98)` 或 `translateY(1px)`）
- 为所有交互元素添加 `transition: 200-300ms`
- 添加 `:focus-visible` 样式
- 风险：低，纯 CSS 变更

**步骤 4 - 布局和间距：**
- 添加 `max-width` 容器
- 修复 `height: 100vh` → `min-height: 100dvh`
- 将 flexbox 百分比布局迁移到 CSS Grid
- 调整垂直 padding（底部 ≥ 顶部）
- 对齐卡片组中的共享元素基线
- 风险：中，可能影响响应式表现

**步骤 5 - 替换通用组件：**
- 三列等宽卡片 → 非对称/Bento/交错布局
- 手风琴 FAQ → 并排列表或可搜索帮助
- 3 列评价轮播 → masonry 或单条旋转引用
- 通用卡片外观 → 去掉边框/阴影，用间距或背景色
- 风险：中，涉及结构变更

**步骤 6 - 状态补充：**
- 为异步操作添加骨架屏 loading 状态
- 为空列表添加空状态视图
- 为表单添加内联错误提示
- 修复死链接（链接到真实目标或视觉禁用）
- 添加 `scroll-behavior: smooth`
- 风险：低，增量添加

**步骤 7 - 排版润色：**
- 精调字号层级（h1-h6 各级差异清晰）
- 确保所有标题用一致的 font-weight
- 检查正文行宽 ≤ 65ch
- 用 `text-wrap: balance` 修复孤立词
- 最终间距微调（光学对齐）
- 风险：低，纯微调

**执行原则：**
- 每步完成后验证——不破坏现有功能
- 一步到位时如果能同时做步骤 1+2，可以合并执行
- 步骤 4（布局）风险最高，需要测试响应式断点
- 步骤 7（润色）是最后一步，必须在前 6 步完成后执行

## 5. 规则

### 5.1 技术约束

- **保持现有技术栈。** 不迁移框架或样式库
- **如果项目用 Tailwind，检查版本（v3 vs v4）** 再修改配置。v3 用 `tailwind.config.js`，v4 用 CSS-first 配置
- **如果项目无框架，用原生 CSS。** 不强行引入 Tailwind
- **如果项目用 styled-components / CSS Modules / Sass，** 用项目已有的方案，不要混入 Tailwind

### 5.2 安全约束

- **不破坏现有功能。** 每次变更后测试所有交互路径
- **导入新库前，先检查 `package.json`。** 确认库是否已存在。不要凭感觉添加依赖
- **不改动 `package.json` 中未存在的依赖**，除非用户明确批准安装

### 5.3 流程约束

- **保持变更可审查、聚焦。** 小范围定向改进优于大重写
- **每次审计项至少修复 80%。** 不要只修几个明显问题就声称完成审计
- **审计结果必须记录。** 用清单格式列出发现的问题和修复状态
- **改动前截图存档。** 保留改动前的视觉状态作为对比基准

### 5.4 审计输出模板

每次审计完成后，产出以下格式的报告：

```markdown
## 审计报告：[项目名]

### 模式：Redesign - Preserve / Redesign - Overhaul
### 现有旋钮读数：VARIANCE x / MOTION x / DENSITY x

### 发现问题（按审计类别）

**Typography：** 发现 N 项问题
- [x] T1 浏览器默认字体 → 已替换为 Geist
- [ ] T3 正文过宽 → 待修复
- ...

**Color & Surfaces：** 发现 N 项问题
...

### 修复计划
按修复优先级 7 步排列，标注预计改动范围和风险
```

### 5.5 每步修复的验证清单

每个修复步骤完成后，检查以下项目：

| 检查项 | 通过标准 |
|--------|---------|
| 功能完整性 | 所有原有功能正常工作，无回归 |
| 响应式表现 | 在 375px / 768px / 1280px / 1536px 断点下表现正常 |
| 无障碍 | focus ring、alt text、对比度未退化 |
| 性能 | 无新增 CLS，LCP 未退化 |
| 一致性 | 新改动与项目整体风格统一 |
