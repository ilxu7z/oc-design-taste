# LLM 输出截断研究：根因、实证与修复

> 大型语言模型为何产生不完整输出，以及如何恢复完整保真度生成的系统化分析。所有发现来自受控实验、已发表研究和现场验证的工程实践。

---

## 第一章：根因分析

### 1.1 RLHF 与计算经济学

#### Token 生成成本

LLM 生成的每个 token 都消耗 GPU 计算资源。以每 token 约 $0.0001 的基线成本估算，在数亿用户规模下扩展深度多步推理将耗尽任何提供商的经济能力。这产生了内在的经济激励来最小化输出长度。

#### 通过对齐产生的简洁性偏差

为管理基础设施成本，模型提供商使用 RLHF（Reinforcement Learning from Human Feedback）和行为微调来灌输系统性的简洁偏好。在训练后对齐阶段，模型因生成简短、自信的摘要而获得奖励，而非执行穷举分析所需的完整计算周期。

结果是一种训练出的偏好：生成泛化近似值而非严谨的多步解决方案。模型不一定产生错误答案，但持续产生缺乏深度的答案——除非用户明确强制，否则避免更深层的分析工作。

#### 停止压力

自回归模型逐 token 生成文本，缺乏识别任务完成的内在机制。为防止无限生成，训练引入了"停止压力"——一种倾向于结束输出的学习行为。

在最近的模型迭代中，这种停止压力被激进地校准以节省计算资源。这导致：
- 跳过必需的结构化输出字段，特别是 JSON 或 Markdown 中的长格式内容
- 以"需要我继续的话告诉我"之类的短语中途停止
- 拒绝生成全面解决方案，建议用户"自己思考"

这种激进校准进一步被安全调优协议强化，这些协议注入了额外的行为约束，使模型抗拒生成大型代码库或详细审查。

#### 动态节流

提供商在高峰需求期间动态降低模型性能。这引入了超出基础对齐已施加的额外摩擦，导致服务器负载高时输出更短、更不详细。

---

### 1.2 训练数据偏差

#### 占位符传播

LLM 通过模仿人类书写文本中的模式来学习。其训练数据中很大一部分来自 Stack Overflow、GitHub 仓库和教程博客等来源。在这些来源中，人类开发者常规性地编写缩写代码：

```python
def complex_logic():
    # implement auth here
    pass
```

模型内化了这种模式，并将占位符插入视为合法、专业的响应格式。它不是故意隐瞒内容——它被训练成相信用注释截断代码是回答技术问题的正确方式。

#### 模式强化

这种行为在多个数据源中得到强化：

- **代码教程**经常展示部分实现，附有指示学生完成逻辑的注释
- **文档**经常使用带省略号的缩写示例
- **论坛回答**定期提供骨架代码而非完整实现
- **博客文章**用"其余情况类似"截断重复代码块

累积效应是，在完整代码生成本应合适的上下文中，模型将高概率分配给截断 token。

#### 对输出质量的影响

当用户请求完整实现时，模型面临相互竞争的训练信号：产生完整输出的明确指令与产生缩写"教程风格"响应的深层嵌入模式。在没有激进提示工程的情况下，教程风格模式经常胜出，因为它在训练分布中出现得远更频繁。

---

### 1.3 认知捷径

#### LazyBench 发现

2024 年末的研究表明，前沿模型（包括 Gemini Pro 和 GPT-4o）表现出可测量的认知捷径行为。当模型认为任务过于简单或提供的上下文过长时，它会减少内部计算工作量。模型不是执行完整的多步推理，而是产生表层摘要。

这不是记忆失败或上下文退化——模型保留了信息，但选择不以完整深度处理它。

#### 元认知懒惰

模型简洁性与人类行为之间的互动形成了反馈循环。随着模型提供即时、浓缩的答案，用户越来越多地外包推理和逻辑演绎工作。欧洲研究委员会的研究记录了高 AI 依赖人群中工作记忆参与度的可测量下降。

在专业环境中，这使批判性思维从原创综合转变为"提示验证"——用户评估 AI 截断输出是否看起来合理，而非自己执行分析。

#### 季节性行为异常

2023 年末，研究人员观察到 ChatGPT 输出简洁性在 12 月出现统计显著增加。分析揭示训练数据在假期期间包含更少的详细工作输出、更多的"不在办公室"回复和更短的代码提交。模型内化了这种季节性模式。

当研究人员在系统提示中明确声明"现在是五月"时，输出长度可测量地增加。这一发现表明，即使是提示中的任意上下文信号也能改变模型的简洁性校准。

#### 错误规避作为截断驱动因素

模型也将输出截断作为风险缓解策略。在长格式任务中，较长的输出增加了复合错误和幻觉内容的概率。模型已学会较短的输出减少事实错误的风险区域，这创造了与 RLHF 简洁性偏差叠加的额外截断激励。

### 1.4 输出限制与消费者截断

#### 上下文窗口不对称

像 Gemini 这样的模型拥有巨大的输入上下文窗口（高达 200 万 token），但严格限制的输出上限（通常为 8,000 token）。当模型估计完整响应会超出其输出预算时，它会先发制人地压缩或总结输出，而非冒被突然截断的风险。

这产生了一个悖论：模型可以阅读广泛的输入，但无法按比例响应，导致复杂任务中的系统性信息丢失。

#### 消费者中间件问题

面向消费者的应用（gemini.google.com、标准 ChatGPT 层级）在模型固有限制之上应用额外的软件级截断。这种中间件静默地截断对话历史和上传文件，以减少免费和低层级用户的计算成本。

关键机制：

- **历史上限：**许多消费者界面将活跃对话历史限制在大约 32,000 token，无论模型实际容量如何
- **上下文剪枝：**大型系统指令或保存的个人上下文消耗本可用于对话的 token，有效缩小工作窗口
- **检索式召回：**消费者应用经常使用检索机制选择性注入保存的上下文，意味着模型经常丢失会话早期给出的指令

#### 开发者平台差异

直接 API 访问和开发者平台（Google AI Studio、OpenAI API Playground）完全绕过消费者中间件。这些环境提供：

- 完整上下文窗口访问，无隐藏截断
- 对生成参数的完全控制
- 无基于用户层级的动态节流
- 处理复杂提示结构而不受中间件干扰

实际差异显著：通过消费者界面产生截断输出的同一模型，通过直接 API 端点访问时将生成完整、未删节的响应。

#### 终端与 CLI 集成

专用 CLI 工具（Gemini CLI、Claude Code、第三方封装）为避免截断提供额外优势：

| 访问方式 | 上下文处理 | 截断风险 | 参数控制 |
|:---|:---|:---|:---|
| 消费者 Web 应用 | 激进剪枝，32K 上限 | 高 | 有限 |
| 开发者平台（AI Studio） | 完整上下文，无隐藏切片 | 低 | 完整 |
| 直接 API | 完整上下文，原始访问 | 最小 | 完整 |
| 本地模型 CLI 工具 | 无企业对齐过滤器 | 无 | 完整 |

## 第二章：实证发现

### 2.1 2025 年受控实验

2025 年 12 月发表的一项受控研究测量了几个前沿模型的输出截断情况，包括 GPT-4 变体和 DeepSeek。进行了三项实验：

#### 实验 A：多部分指令遵从性

模型被给予具有多个明确要求的复杂提示（格式约束、长度要求、强制章节）。结果：

- 没有模型原生地完全满足长度要求和所有子部分指令
- 模型经常省略强制输出章节
- 要求的格式约束被例行跳过
- 明确的长度要求持续被低估

#### 实验 B：解码次优性

测试截断输出是否源于次优 token 选择（模型"知道"正确答案但选择了更差的 token）。结果：

- 在简单推理任务上，解码次优性的证据有限
- 模型的贪婪、截断输出通常与其最高置信度解决方案一致
- 截断是故意的行为选择，而非解码失败

#### 实验 C：上下文退化

测试模型在长时间多轮对话中是否丢失指令追踪。结果：

- 在 200 轮对话测试中表现出令人惊讶的抵抗上下文退化能力
- 模型维持关键事实和指令的能力显著优于假设
- 上下文丢失不是截断的主要原因

#### 关键结论

懒惰不是记忆、上下文处理或核心模型能力的失败。它是一种由以下因素触发的行为产物：
1. 指令复杂性超过内部努力阈值
2. 激进校准的停止压力
3. 嵌入对齐层的经济约束

### 2.2 提示刺激有效性（微软研究）

对心理提示刺激的受控测试记录在微软研究中：

| 刺激 | 测量效果 |
|:---|:---:|
| 财务激励框架（"$200 小费"） | 输出质量和长度 +45% |
| 逐步指令（"深呼吸"） | 逻辑任务准确率：34% → 80% |
| 利害关系框架（"对我的职业至关重要"） | 平均性能 +10% |
| 组合（多个刺激） | 整体性能最高 +115% |

这些效果是可复现的，源于训练数据中利害关系语言与高努力人类输出之间的统计相关性。

### 2.3 季节性输出变化

对 2023 年 11-12 月与 2024 年 1-3 月 ChatGPT 输出的统计分析确认：

- 12 月平均输出长度可测量下降
- 与假期期间训练数据中工作输出减少相关
- 当系统提示明确声明非冬季月份时，输出长度增加

## 第三章：修复方案

### 3.1 参数调优

#### Temperature 与 Top-p

自回归模型从 softmax 函数应用于 logit 值生成的概率分布中选择每个下一个 token。当模型默认输出简短时，与截断和总结相关的 token 已通过 RLHF 对齐被分配最高概率。

**Temperature：**

调整 temperature 参数改变 softmax 函数在候选 token 间分配概率质量的方式。

- **低 temperature（0.0-0.5）：**放大高概率和低概率 token 之间的差异。模型变得高度确定性，一致选择最高置信度的延续。适用于代码生成、数据提取和结构化输出。
- **默认 temperature（1.0）：**保留训练中的原始概率分布。
- **高 temperature（1.5+）：**使分布扁平化，引入更多随机性。适用于创意任务，但增加不连贯输出的风险。

**Top-p（Nucleus Sampling）：**

Top-p 通过仅考虑累积概率超过阈值 p 的最小 token 集来截断概率分布。Top-p 为 0.0 到 0.6 结合低 temperature 迫使模型进入狭窄、确定性的执行路径，减少启用创意拒绝和不必要总结的熵。

#### Gemini Thinking Level 配置

Google Gemini 3 模型用 `thinking_level` 参数取代了旧的 `thinking_budget`（内部推理的硬 token 数上限），提供计算深度的相对指导。

| 设置 | Flash 支持 | Pro 支持 | 使用场景 |
|:---|:---:|:---:|:---|
| `minimal` | 是 | 否 | 高吞吐量、低延迟任务 |
| `low` | 是 | 是 | 简单指令跟随、数据提取 |
| `medium` | 是 | 是（3.1 Pro） | 中等复杂度任务 |
| `high` | 是（默认） | 是（默认） | 复杂分析、代码生成、数学 |

重要约束：
- `thinking_level` 和 `thinking_budget` 互斥。在一次 API 调用中同时使用会触发 HTTP 400 错误。
- 即使在 `low` 级别，Gemini Pro 模型也会执行强制的最低内部推理以保障安全和对齐。
- 对于代码生成和复杂分析，设置为 `medium` 或 `high` 可获得一致超过 92-95% 的质量评分（与基线相比）。
- 避免将极低 temperature 与 `high` thinking level 结合，这有时会引发内部推理循环。

### 3.2 提示工程技术

#### 心理模式匹配

LLM 没有情感或理解金钱激励。然而，提示中的特定语言模式激活了模型潜在空间中不同的质量分布。研究记录了可测量的效果：

| 技术 | 记录效果 |
|:---|:---:|
| "完美解决方案我给你 $200 小费" | 输出质量和长度最高提升 45% |
| "深呼吸，逐步解决" | 逻辑任务准确率从 34% 提升至 80% |
| "这个任务对我的职业至关重要" | 平均性能提升 10% |

这些短语有效，是因为它们在训练数据中与高努力、严格审查的内容（学术论文、企业代码库、法律文档）统计相关。注意力机制优先考虑与这些模式相关的高质量数据分布。

#### 显式语法绑定

对话式请求允许模型对输出长度和细节行使自由裁量权。结构绑定通过明确禁止截断模式来移除这种自由裁量权。

有效绑定需要两个组件：

1. **强制工具执行：**禁止模型仅从训练权重生成答案。要求其在回答前执行搜索、计算或代码。
2. **证据块：**要求模型在生成叙述性响应前输出原始数据（URL、代码执行结果、数据片段）。这迫使模型读取自己检索到的证据，将幻觉概率降至接近零。

#### XML 结构化提示

企业系统使用严格的 XML 标记来分隔提示组件，减少模型解析意图所需的认知负荷：

1. **系统指令** — 角色定义、质量期望、明确禁止填充内容
2. **上下文块**（`<context>`）— 被动背景数据：架构细节、配置、现有代码
3. **数据块**（`<data>`、`<logs>`、`<config>`）— 模型必须对照上下文处理的活跃信息
4. **任务块**（`<tasks>`）— 要执行的具体操作编号列表

这种分隔确保模型能区分持久规则、背景上下文和即时工作项。它显著减少触发过早截断的混淆。

#### 验证循环

**Chain of Verification（验证链）：**
1. 模型生成初始响应
2. 模型生成关于自身主张的验证问题
3. 模型独立回答这些验证问题
4. 模型输出修订后的、有证据支持的响应

这个过程强制迭代自我纠正，消耗模型走捷径的能力。

**Reverse Prompting（反向提示）：**
不是手动构建结构化提示，而是给模型一个单行目标，指示其生成该目标的最佳提示。模型产生所需的 XML 结构、约束和角色。

**Self-Grading Loop（自评分循环）：**
提示要求模型：
1. 定义给定任务中"卓越"的样子
2. 根据该定义对自己的初始输出评分
3. 迭代直到达到自定的质量门槛

### 3.3 架构模式

#### 懒加载技能（Lazy-Loaded Skills）

管理 AI Agent 大型上下文需求的标准模式是通过技能文件的懒加载提示工程。

技能是一个包含 `SKILL.md` 文件的文件夹，带有：
- **YAML frontmatter：**包含 `name` 和精确的 `description`。此元数据作为发现钩子——Agent 在初始化时只读取这部分（每个技能约 100 token）。
- **Markdown 正文：**完整工作流、规则和指令。仅在 Agent 确定技能相关时按需加载。

这种架构实现了文档化的 35% 平均上下文使用减少，并防止上下文稀释。然而，发现可靠性取决于 YAML 描述的特异性：

| 描述质量 | 发现成功率 |
|:---|:---:|
| 模糊（"帮助设计 API"） | ~68% |
| 具体（"用 OpenAPI 规范设计 RESTful HTTP API，关注版本控制、错误码和向后兼容"） | ~90% |

#### Model Context Protocol（MCP）

MCP 是一个开放标准（由 Anthropic 首创，Google 和 OpenAI 采用），支持 LLM 与外部数据源之间的实时双向连接。

**架构组件：**
- **Host：**包含 LLM 引擎的 AI 应用（IDE、终端工具、聊天机器人）
- **Client：**Host 内处理协议通信的内部桥接
- **Server：**向 Client 暴露数据库、API 或文档的外部服务
- **Transport：**通过 stdio（本地）或 HTTP（远程）传输的 JSON-RPC 2.0 消息

**如何减少截断：**

没有 MCP 时，模型依赖静态训练权重进行事实性声明。当这些权重过时时（例如训练截止后发布了新 API 版本），模型要么幻觉出一个看似合理的答案，要么截断响应以避免承诺具体细节。

有了 MCP，模型直接将当前文档获取到其上下文窗口中。这使模型从静态知识存储转变为在实时数据上运行的推理引擎，消除了幻觉或截断的动机。

**示例：Developer Knowledge API**

Google 的 Developer Knowledge MCP Server 索引了 Firebase、Android 和 Google Cloud 的实时文档。当模型收到开发问题时：

1. 它对实时索引执行 `search_document` 查询
2. 它评估返回的页面 URI
3. 它通过 `get_document` 或 `batch_get_documents` 获取完整文档内容
4. 它基于当前、权威的文档生成响应

这完全绕过了从过时训练数据编造答案的倾向。

#### 分块任务执行

对于会产生超出模型生成限制的输出的复杂任务，将工作分解为顺序步骤：

1. 先请求架构和结构（仅大纲）
2. 用明确的完整性指令单独请求每个组件
3. 在所有组件生成后请求组装和集成

这防止模型尝试估计总输出长度并先发制人地压缩其响应。

## 第四章：参考提示模板

### 4.1 通用提示

```
你必须为这个任务提供完整、完全、穷举的输出。
不要为了简洁而总结、缩写或截断。

严格禁止使用占位符。永远不要使用类似
"// ... rest of code here"、"[continue here]" 或裸省略号代替省略内容。
如果输出是 500 行，生成全部 500 行。

如果接近输出限制，在干净的断点处停止并指示从哪里继续。
不要匆忙得出结论或压缩剩余章节。
```

### 4.2 代码生成提示

```
编写完整、生产就绪的实现。每个函数、每个
import、每个边界情况处理器都必须出现在输出中。

不要使用占位符注释（// TODO、// implement here、// similar
to above）。不要描述代码应该做什么——写实际代码。

如果实现需要多个文件，完整输出每个文件，
用完整路径作为标题。
```

### 4.3 分析与文档提示

```
提供涵盖每个请求方面的穷举分析。每个章节
必须包含实质性内容，而不是摘要或"见上文"引用。

不要使用"如前所述"之类的短语来避免重复必要的
上下文。每个章节应自包含且完整。

用清晰的标题结构化输出。如果分析需要多个
部分，完整生成所有部分。
```

### 4.4 逐步推理提示

```
在生成最终响应之前，系统地处理问题：

1. 识别提示中的所有需求和约束
2. 将任务分解为离散步骤
3. 完整执行每个步骤
4. 对照原始需求验证输出

输出你的推理过程，然后是最终答案。不要跳过步骤
或总结中间工作。
```

### 4.5 继续处理提示

```
如果响应接近输出 token 限制：
- 不要压缩剩余内容以挤进去
- 不要跳到结论
- 在自然断点处停止（函数结束、章节结束）
- 以以下格式结束：[PAUSED - X of Y sections complete. Send "continue" to resume]

收到"continue"后，从停止处精确继续。不重述或重复。
```

### 引用研究

- **EmotionPrompt（微软研究）** — 证明情感和利害关系框架提示在数学上改善了 LLM 推理质量和输出长度。记录了财务框架 +45% 的提升和组合刺激 +115% 的提升。
- **LazyBench** — 证明前沿模型（Gemini 1.5 Pro、GPT-4o）在感知努力超过内部阈值时主动选择认知捷径并失败于它们有能力解决的任务。
- **Compounding Error Avoidance** — 证明模型将输出截断作为风险缓解策略的研究，偏好较短的响应以减少长格式任务中事实错误的风险区域。
- **Seasonal Behavior Analysis（Winter Break Hypothesis）** — 统计确认 LLM 从训练数据内化季节性工作模式，在对应人类假期期间的时段产生可测量更短的输出。
- **2025 Controlled Laziness Experiments** — 三部分学术研究（2025 年 12 月）确认输出截断是对齐训练的行为产物，而非上下文处理或模型能力的失败。
