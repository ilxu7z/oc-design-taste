# 设计批评方法论

> 结构化的设计审查框架。不是"我觉得不好看"，而是可量化、可追踪、可比较的评估系统。

---

## 1. 批评流程

```
确定目标（文件/URL/组件）
  ↓
视觉审查（设计总监视角）
  ↓
认知负荷评估
  ↓
角色走查（2-3 个角色）
  ↓
输出结构化报告
  ↓
优先排序 + 建议动作
```

### 1.1 确定目标

将模糊描述转化为具体文件路径或 URL：
- "首页" → `pages/index.astro` 或 `index.html`
- "设置弹窗" → 对应组件文件
- "这个页面" → 当前 URL 或源文件

### 1.2 视觉审查

像设计总监一样审视：

| 维度 | 检查内容 |
|------|---------|
| **AI 痕迹** | 检查 SKILL.md §5 痕迹目录和 §4.I Absolute Bans |
| **整体设计** | 层级、信息架构、情感匹配、可发现性、构图、排版、色彩、无障碍、状态、文案、边缘情况 |
| **认知负荷** | 见 §2 认知负荷评估 |
| **情感旅程** | peak-end 规则、情绪低谷、高风险时刻的安慰 |
| **Nielsen 启发式** | 见 §3 启发式评分 |

输出：AI 痕迹判定、启发式分数、认知负荷、情感旅程、2-3 个优点、3-5 个优先问题、角色走查红旗、次要观察、建议性问题。

### 1.3 输出结构化报告

#### 设计健康评分
Nielsen 10 启发式评分表（见 §3）。

#### 反模式判定
- 整体美学感觉、布局同质化、通用构图、错失的个性机会
- SKILL.md §5 痕迹 + §4.I 禁令的命中情况

#### 总体印象
简要直觉：什么有效、什么无效、最大的单一机会。

#### 优点（2-3 个）
具体说明为什么有效。

#### 优先问题（3-5 个）
每个问题标注 P0-P3 严重性（见 §4）：
- **[P?] 问题**：清楚命名
- **为什么重要**：如何伤害用户
- **修复方案**：具体可执行的建议
- **建议命令/Domain**：可参考的 SKILL.md 路由

#### 角色走查红旗
见 §5 角色走查。

#### 次要观察
值得注意的较小问题。

#### 建议性问题
可能解锁更好方案的挑衅性问题。

## 2. 认知负荷评估

认知负荷 = 使用界面所需的总心理努力。过载用户会犯错、沮丧、离开。

### 2.1 三种认知负荷

| 类型 | 含义 | 应对策略 |
|------|------|---------|
| **内在负荷** | 任务本身固有复杂度 | 分解步骤、提供脚手架（模板/默认/示例）、渐进展示 |
| **外在负荷** | 糟糕设计制造的额外负担 | **无情消除**——这是纯浪费 |
| **关联负荷** | 学习所需的努力 | 这是**好的**认知负荷——渐进展示 + 一致模式奖励学习 |

### 2.2 外在负荷常见来源

| 来源 | 症状 | 修复 |
|------|------|------|
| 混乱导航 | 用户需要心理映射才能找到目标 | 清晰导航 + 当前位置指示（面包屑/active 状态） |
| 模糊标签 | 用户被迫猜测含义 | 用用户语言，不用术语 |
| 视觉杂讯 | 每个元素视觉权重相同 | 建立清晰层级 |
| 不一致模式 | 同一行为在不同位置表现不同 | 标准化交互模式 |
| 不必要步骤 | 用户意图和结果之间有多余操作 | 减少中间步骤 |
| 记忆桥梁 | 用户必须记住前一步的信息才能完成后一步 | 保持相关上下文可见 |

### 2.3 认知负荷检查清单

- [ ] **单一焦点**：用户能不受干扰地完成主要任务？
- [ ] **分块**：信息以可消化的组呈现（≤4 项/组）？
- [ ] **分组**：相关项目视觉上聚合（邻近/边框/共享背景）？
- [ ] **视觉层级**：页面上什么最重要一目了然？
- [ ] **一次一件事**：用户能一次专注一个决策？
- [ ] **最少选择**：决策简化（≤4 个可见选项/决策点）？
- [ ] **工作记忆**：用户不需要记住上一屏的信息才能在当前屏操作？
- [ ] **渐进展示**：复杂度只在需要时揭示？

**评分**：失败项计数。0-1 = 低（好）。2-3 = 中等（需尽快处理）。4+ = 高（关键修复）。

### 2.4 工作记忆规则

人类工作记忆同时保持 ≤4 个项目（Miller 法则，Cowan 2001 修订）。

| 可见选项数 | 评估 |
|-----------|------|
| ≤4 | 在工作记忆限制内，可控 |
| 5-7 | 边界；考虑分组或渐进展示 |
| 8+ | 过载；用户会跳过、误点、放弃 |

**实际应用**：
- 导航菜单：≤5 顶级项目
- 表单组：≤4 个可见字段/组
- 操作按钮：1 主 + 1-2 次，其余放菜单
- Dashboard 小部件：≤4 关键指标不滚动
- 定价档位：≤3 选项（更多 = 分析瘫痪）

## 3. Nielsen 启发式评分指南

对 10 个可用性启发式分别打 0-4 分。要诚实：4 分 = 真正优秀，不是"还行"。

| # | 启发式 | 含义 |
|---|--------|------|
| 1 | 系统状态可见性 | 通过及时适当的反馈让用户了解发生的情况 |
| 2 | 系统与现实世界的匹配 | 用用户的语言，遵循现实世界的约定 |
| 3 | 用户控制与自由 | 用户需要从不需要的状态中有清晰的"紧急出口" |
| 4 | 一致性与标准 | 用户不应疑惑不同词语/情境/动作是否意味着相同 |
| 5 | 错误预防 | 比好的错误信息更好的是能预防错误的设计 |
| 6 | 识别而非回忆 | 最小化记忆负荷，让对象/动作/选项可见或易检索 |
| 7 | 使用灵活性与效率 | 加速器（键盘快捷键等）让专家更快 |
| 8 | 美学与极简主义 | 界面不应包含无关或很少需要的信息 |
| 9 | 帮助用户识别/诊断/恢复错误 | 错误信息用通俗语言，精确指出问题并建设性建议解决方案 |
| 10 | 帮助与文档 | 帮助应该容易找到、任务聚焦、简洁 |

### 评分标准

每个启发式的打分标准：

| 分数 | 含义 |
|------|------|
| 0 | 没有反馈；用户在猜测 |
| 1 | 罕见反馈；大多数操作无可见响应 |
| 2 | 部分；一些状态被传达，重大缺口仍存在 |
| 3 | 良好；大多数操作给出清晰反馈，轻微缺口 |
| 4 | 优秀；每个动作确认，进度始终可见 |

**总分范围**：0-40 分

| 分数范围 | 评级 | 含义 |
|---------|------|------|
| 36-40 | 优秀 | 只需微调 |
| 28-35 | 良好 | 处理弱点区域，基础扎实 |
| 20-27 | 可接受 | 需要显著改进 |
| 12-19 | 差 | 需要重大 UX 重构 |
| 0-11 | 危急 | 需要重新设计；当前状态不可用 |

**大多数真实界面得分 20-32。** 给自己打 36+ 要有真正出色的理由。

## 4. 问题严重性分级（P0-P3）

| 优先级 | 名称 | 描述 | 动作 |
|--------|------|------|------|
| **P0** | 阻塞 | 完全阻止任务完成 | 立即修复；这是障碍 |
| **P1** | 重大 | 导致显著困难或困惑 | 发布前修复 |
| **P2** | 次要 | 烦人但有替代方案 | 下次迭代修复 |
| **P3** | 打磨 | 锦上添花，无实际用户影响 | 有时间就修复 |

**判断标准**：如果不确定在两个级别之间，问："用户会因为这个问题联系客服吗？"如果是 → 至少 P1。

## 5. 角色走查

通过 5 个不同用户原型的眼睛测试界面。每个角色暴露不同的失败模式。

**使用方法**：选择 2-3 个与被批评界面最相关的角色。以每个角色身份走完主要用户操作。报告**具体红旗**，不是泛泛关注。

### 5.1 角色清单

| 角色 | 代号 | 特征 | 测试重点 |
|------|------|------|---------|
| **不耐烦的超级用户** | Alex | 专家，追求效率，讨厌手把手 | 键盘快捷键、批量操作、跳过引导、Esc 关闭 |
| **困惑的首次使用者** | Jordan | 从未使用过此类产品，需要每步引导 | 首次操作是否 5 秒内明确、图标有无标签、术语是否可理解 |
| **无障碍依赖用户** | Sam | 屏幕阅读器/键盘导航，可能低视力/运动障碍 | 纯键盘能否完成流程、焦点指示器可见、ARIA 标签、颜色对比度 ≥4.5:1 |
| **故意的压力测试者** | Riley | 故意在边界场景测试（空状态/长字符串/特殊字符/刷新中流程/多标签页） | 边缘情况、错误恢复、空状态设计、工作流中断 |
| **分心的移动端用户** | Casey | 单手操作、频繁被打断、可能慢速网络 | 拇指热区、状态保留、轻量资产、触摸目标 ≥44×44pt |

### 5.2 角色选择指南

| 界面类型 | 首选角色 |
|---------|---------|
| Landing Page / 营销 | Jordan, Riley, Casey |
| Dashboard / 后台 | Alex, Sam |
| 电商 / 结账 | Casey, Riley, Jordan |
| 引导流程 | Jordan, Casey |
| 数据密集 / 分析 | Alex, Sam |
| 表单密集 / 向导 | Jordan, Sam, Casey |

### 5.3 走查输出格式

```
**[角色代号]（[角色名]）**：
- [具体操作] → [发现的红旗]
- [具体元素] → [为什么失败]
- [建议修复]
```

**不要写泛泛的角色描述，要写具体什么坏了。**

## 6. 与 SKILL.md 的集成

设计批评是 Domain 路由的一个**操作模式**，不是独立 Domain。

| 批评触发 | 推荐动作 |
|---------|---------|
| 用户说"帮我看看这个设计" | 走完整批评流程（§1） |
| 用户说"审查一下这个页面" | 走完整批评流程 |
| 子 Agent 交付产出后验收 | 轻量批评：AI 痕迹 + Pre-Flight Check |
| 设计评审会议 | 走 §3 启发式评分 + §5 角色走查 |

**批评流程不修改代码**——只输出报告和建议。用户决定后续动作。
