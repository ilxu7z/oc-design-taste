# 设计 Token 提取流程

> 从现有代码库识别可复用模式、组件和设计变量，将其提取并固化到设计系统中。

---

## 1. 发现设计系统

找到项目中已有的设计系统、组件库或共享 UI 目录。理解其结构：组件组织、命名规范、Token 结构、导入/导出约定。

**⚠️ 如果项目没有设计系统**，先确认用户的首选位置和结构，再创建。不要自作主张。

## 2. 识别提取机会

扫描目标区域，寻找以下模式：

| 类型 | 信号 | 提取条件 |
|------|------|---------|
| **重复组件** | 类似 UI 模式出现 3+ 次 | 相同意图，非视觉巧合 |
| **硬编码值** | 颜色/间距/排版/阴影散落在文件中 | 应该是 token |
| **不一致变体** | 同一概念有多种实现 | 统一为组件变体 |
| **组合模式** | 布局/交互模式重复出现（表单行、工具栏组、空状态） | 提取为模式组件 |
| **Type 风格** | 重复的 font-size + weight + line-height 组合 | 提取为 typography scale |
| **动画模式** | 重复的 easing/duration/keyframe | 提取为 motion token |

**评估原则**：只提取使用 3+ 次且意图相同的模式。过早抽象比重复更糟糕。

## 3. 规划提取

| 规划项 | 说明 |
|--------|------|
| **组件提取** | 哪些 UI 元素成为可复用组件？ |
| **Token 创建** | 哪些硬编码值成为设计 token？ |
| **变体支持** | 每个组件需要什么变体？ |
| **命名规范** | 组件名/token 名/prop 名匹配现有模式 |
| **迁移路径** | 如何重构现有使用以消费新共享版本？ |

**设计系统是增量的。** 只提取现在明确可复用的，不提取所有"可能"可复用的。

## 4. 提取与增强

为每种类型构建改进的、可复用的版本：

### 4.1 组件
- 清晰的 props API + 合理默认值
- 适当的变体（不同用例）
- 无障碍内置（ARIA、键盘导航、焦点管理）
- 文档和使用示例

### 4.2 设计 Token
- 清晰命名（原始 vs 语义）
- 适当的层次和组织
- 每个 token 的使用文档

### 4.3 模式
- 何时使用此模式
- 代码示例
- 变体和组合

## 5. 迁移

| 步骤 | 动作 |
|------|------|
| 5.1 | 搜索所有实例 |
| 5.2 | 系统性替换为共享版本 |
| 5.3 | 测试视觉和功能等价 |
| 5.4 | 删除旧实现（死代码禁止残留） |

## 6. 文档更新

- 将新组件添加到组件库文档
- 记录 token 使用和值
- 添加示例和指南
- 更新任何 Storybook 或组件目录

---

## 7. 红线（NEVER）

| 禁止 | 原因 |
|------|------|
| 提取没有通用化的上下文特定实现 | 过早抽象 |
| 创建过于通用以至于无用的组件 | 每个场景都"适合"= 没有场景真正需要 |
| 忽略现有设计系统规范 | 不一致比不存在更糟糕 |
| 跳过 TypeScript 类型或 prop 文档 | 无类型 = 无约束 |
| 为每个单一值创建 token | token 应该有语义含义 |
| 提取意图不同的模式 | 两个按钮看起来相似但用途不同 → 保持分开 |

---

## 8. Token 命名规范参考

### 原始 Token（Primitive）
```
--color-blue-500
--spacing-4
--radius-md
--font-size-sm
--shadow-lg
--ease-standard
--duration-fast
```

### 语义 Token（Semantic）
```
--color-primary: var(--color-blue-500)
--color-surface: var(--color-blue-50)
--color-danger: var(--color-red-600)
--spacing-section: var(--spacing-8)
--radius-card: var(--radius-md)
```

### 层次规则
- 原始 token 只引用值
- 语义 token 引用原始 token
- **禁止反向引用**（原始不应引用语义）
- 暗色模式只重定义语义层
