# 视觉自进化引擎 v2

> **目标**：让团队的视觉输出能力螺旋上升，不依赖人工发现规则。
> **原则**：每次视觉任务都是一次实验——产出对比预期，差距驱动进化。
> **与 SKILL.md 的关系**：SKILL.md 是推理引擎（静态基线），本文件是进化协议（动态引擎）。
> **v2 变化**：适配 v6 模块化架构——进化不再只作用于 SKILL.md，而是作用于整个文件体系（domains/profiles/libraries/knowledge）。

---

## 1. 架构概览

```
视觉任务触发
  ↓
Layer 1: 任务闭环（每次任务）
  读取基线 → Domain 路由 → 加载文件 → 执行 → Pre-Flight → 产出 → 复盘
  ↓
Layer 2: 差距蒸馏（跨任务）
  复盘 → 差距归因 → 提炼策略 → 归档到正确位置
  ↓
Layer 3: 规则固化（低频大更新）
  策略验证 N 次 → 升级到对应文件（SKILL.md/domain/profile/library/knowledge）
  或：规则在实战中总是被突破 → 降级/删除
```

**三层各自有独立的触发条件和数据结构，不互相阻塞。**

**v2 关键变化：** 规则不再只固化到 SKILL.md。进化引擎可以更新：
- `SKILL.md` — 核心纪律（痕迹目录、Pre-Flight Check）
- `domains/*.md` — 领域特化规则
- `profiles/*.md` — 美学预设
- `libraries/*.md` — 组件/模式库
- `knowledge/*.md` — 知识层

## 3. Layer 1: 任务闭环（每次任务）

### 3.1 任务前：读取基线

1. 读 SKILL.md v6（Section 0-6 核心引擎）
2. 根据 Design Read 路由到正确 Domain 文件，加载
3. 如需美学预设，加载对应 Profile 文件
4. 读 `visual-prompt-strategies.md` 中匹配当前 Domain 的策略
5. 如有该项目的历史复盘记录，读最近 2-3 份

### 3.2 任务后：强制复盘

**可以跳过的任务：**
- 微调（改一个颜色值、调一行间距）
- 纯技术修复（bug 修复、响应式调整）
- 用户明确说"就这样"

**其他所有视觉任务必须复盘**，产出物交付前/交付时执行：
1. 产出 vs 预期对比（自然语言，不是 checklist 打勾）
2. 差距归因（6 个类别，v2 新增"路由错误"）
3. Domain 路由评估——加载的 domain 文件是否正确？是否需要新建 domain？

---

## 4. Layer 2: 差距蒸馏（跨任务）

### 4.1 v2 扩展：策略 → 文件路由

```
这个差距以前出现过吗？
  ↓
没有 → 新问题，记录复盘，等待积累
  ↓
出现 ≥ 2 次 → 触发蒸馏
  ↓
蒸馏产出 → 判断应该写入哪个文件：

| 策略性质 | 写入位置 |
|---------|---------|
| 核心纪律（所有 domain 共用） | SKILL.md Section 4/5/6 |
| Domain 特化规则 | domains/{domain}.md |
| 美学预设调整 | profiles/{profile}.md |
| 新组件/模式 | libraries/component-arsenal.md 或 motion-patterns.md |
| 背景知识 | knowledge/{topic}.md |
```

### 4.2 v2 扩展：Domain 创建

如果差距反复指向"没有对应的 domain 文件"（≥ 3 次），考虑新建 domain：
1. 评估该场景是否足够独立（不是现有 domain 的子集）
2. 如果是，创建 `domains/{new-domain}.md`
3. 在 SKILL.md Section 2 Domain 路由表中注册

### 4.3 Prompt 策略生命周期

```
新建（实验中）
  ↓ 被使用 ≥ 3 次
已验证
  ↓ 被使用 ≥ 5 次 且 效果稳定
固化候选
  ↓ 审查 → 写入对应文件
已固化
  ↓ 实战中 >50% 被突破条件触发
降级回策略库（标注来源文件和原位置）
```

---

## 5. Layer 3: 规则固化（低频大更新）

### 5.1 触发条件

1. 策略验证 ≥ 5 次 → 写入对应文件
2. 差距热力图某类问题 >10 次未改善
3. 新 AI 痕迹模式出现 ≥ 3 次
4. 子 Agent 屡次违反某条规则（规则描述不清晰）
5. **v2 新增：** 发现某个 domain 文件内容不足或过载（>1000 行）

### 5.2 固化到正确位置

v1 的固化只写入 SKILL.md。v2 的固化写入**正确的文件**：
- 核心纪律 → SKILL.md
- 领域规则 → 对应 domain 文件
- 组件模式 → 对应 library 文件
- 知识 → 对应 knowledge 文件

**避免 SKILL.md 膨胀。** SKILL.md 是推理引擎，不是规则仓库。

### 5.3 降级流程

规则如果 >50% 被突破条件触发 → 从源文件移除，放回策略库，标注：
```
状态: 降级
来源文件: domains/web-design.md §4.2
降级原因: 实战中 [场景] 下频繁不适用
降级日期: YYYY-MM-DD
```

## 6. 归因模型

### 6.1 六大归因类别（v2 新增"路由错误"）

| 类别 | 含义 | 典型信号 |
|------|------|---------|
| **品味判断** | 旋钮值不对、美学方向偏离 | "不够高端""太花""不像品牌" |
| **Prompt 策略** | 给生图模型/AI 的指令不够精确 | 生图输出差一点、子 Agent 产出偏离 |
| **技术执行** | 代码/CSS/HTML 实现有问题 | 响应式坏掉、动效卡顿、排版错位 |
| **素材质量** | 图片/图标/占位图选择不当 | 占位图风格不匹配、图标不够专业 |
| **用户沟通** | 需求理解偏差 | "不是我要的""方向搞错了" |
| **Domain 路由** | 加载了错误的 domain 文件或缺失文件 | "这个场景的规则不够详细""Domain 规则不适用" |

### 6.2 归因 → 行动映射

| 归因 | Layer 2 行动 | Layer 3 行动（持续时） |
|------|-------------|----------------------|
| 品味判断 | 优化旋钮预设表 | 更新 SKILL.md Section 1 场景预设 |
| Prompt 策略 | 新建/优化策略 | 固化到对应 domain 文件 |
| 技术执行 | 记录技术笔记 | 更新对应 domain 文件 |
| 素材质量 | 优化生图 prompt / 资产优先级 | 更新 domains/image-generation.md |
| 用户沟通 | 优化 Section 0 信号读取 | 更新 SKILL.md Section 0 |
| Domain 路由 | 调整路由决策 | 更新路由表 / 新建 domain |

## 7. 进化节奏

| 频率 | 行动 |
|------|------|
| **每次任务** | Layer 1 强制复盘（除微调/修复外） |
| **每周** | Layer 2 批量蒸馏，更新差距热力图 |
| **每月** | Layer 3 规则固化审查，更新文件体系 |
| **每季度** | 全面审查：SKILL.md + 全部 domains/profiles/libraries/knowledge + 策略库 + 热力图。评估整体趋势。清理过时规则。合并重复策略。调整进化引擎参数。 |

## 8. 文件体系维护规则

### 8.1 SKILL.md

- **目标行数：** 400-600 行（推理引擎，不膨胀）
- **如果超过 700 行：** 检查是否有规则应该移到 domain/knowledge 文件
- **只保留：** 推理逻辑（Brief/旋钮/路由）、核心纪律（排版/色彩/布局/文案）、痕迹目录、Pre-Flight Check

### 8.2 domains/

- **目标行数：** 每个 domain 200-600 行
- **如果超过 800 行：** 拆分为子文件或提取到 library
- **新增条件：** 复盘中出现 ≥ 3 次独立场景需求

### 8.3 profiles/

- **原则：** 完整保留原版内容，不压缩
- **修改条件：** 实战中验证了特定参数调整（如颜色值、间距值）

### 8.4 libraries/

- **原则：** 可复用模式库，按需加载
- **更新条件：** 新增验证过的组件模式或代码骨架

### 8.5 knowledge/

- **原则：** 背景知识，不直接执行
- **更新条件：** 新增有实证支持的知识条目
