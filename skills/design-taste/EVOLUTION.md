# 视觉自进化引擎 v1

> **目标**：让团队的视觉输出能力螺旋上升，不依赖人工发现规则。
> **原则**：每次视觉任务都是一次实验——产出对比预期，差距驱动进化。
> **与 SKILL.md 的关系**：SKILL.md 是可执行规则手册（静态基线），本文件是进化协议（动态引擎）。

---

## 1. 架构概览

```
视觉任务触发
  ↓
Layer 1: 任务闭环（每次任务）
  读取基线 → 执行 → 产出 → 对比预期 → 强制复盘
  ↓
Layer 2: 差距蒸馏（跨任务）
  复盘 → 差距归因 → 提炼 prompt 策略 → 归档
  ↓
Layer 3: 规则固化（低频大更新）
  prompt 策略验证 N 次 → 升级为 SKILL.md 规则
  或：规则在实战中总是被突破 → 降级/删除
```

**三层各自有独立的触发条件和数据结构，不互相阻塞。**

## 2. 数据结构

### 2.1 任务复盘记录

每次视觉任务完成后（无论画册/官网/生图/排版），在项目目录下创建或追加：

## 3. Layer 1: 任务闭环（每次任务）

### 3.1 任务前：读取基线

1. 读 SKILL.md 当前版本（Section 1 旋钮 + Section 7 痕迹目录）
2. 读 `shared-knowledge/visual-prompt-strategies.md` 中与本次任务类型匹配的策略
3. 如有该项目的历史复盘记录，读最近 2-3 份

### 3.2 任务中：执行

按 SKILL.md 流程执行。**关键**：如果使用了 prompt 策略库中的策略，在任务记录中标注。

### 3.3 任务后：强制复盘

**这是 Layer 1 的核心。** 每次视觉任务完成后，产出物交付前/交付时，执行：

1. **产出 vs 预期对比** — 不是 checklist 打勾，而是用自然语言描述差距
2. **差距归因** — 把每个差距归入 5 个类别之一：
   - **品味判断** — 调旋钮/选美学方向时的决策偏差
   - **Prompt 策略** — 写给生图模型/子 Agent 的 prompt 不够好
   - **技术执行** — HTML/CSS/代码实现偏差
   - **素材质量** — 生图输出质量、占位图选择、图片处理
   - **用户沟通** — 需求理解偏差、没问关键问题

3. **记录** — 写入项目目录 `visual-review-{日期}.md`

### 3.4 什么情况可以跳过复盘

- 微调任务（改一个颜色值、调一行间距）
- 纯技术修复（bug 修复、响应式调整）
- 用户明确说"就这样，不用改了"

其他所有视觉类任务**必须复盘**。

## 4. Layer 2: 差距蒸馏（跨任务）

### 4.1 单次蒸馏

每次复盘后，判断：

```
这个差距以前出现过吗？
  ↓
没有 → 新问题，记录到 visual-review，等待积累
  ↓
出现 ≥ 2 次 → 触发蒸馏
```

**蒸馏产出**：一条 Prompt 策略（Layer 2.2 格式），写入 `shared-knowledge/visual-prompt-strategies.md`

**蒸馏不是改规则**，是改**怎么做**。区别：

| 改规则（Layer 3） | 改策略（Layer 2） |
|-----------------|-----------------|
| "禁止使用 AI 紫色渐变" | "生成紫色渐变场景时，prompt 加 'muted, desaturated purple tones, no neon glow'" |
| "Hero 标题 ≤ 2 行" | "写 Hero 标题时，先用 6 个词以内写一遍，如果表达不清楚再扩展到 2 行" |
| 适用于所有项目 | 适用于特定场景 |

### 4.2 批量蒸馏

每完成 5-10 次视觉任务（或每周一次）：

1. 读所有 `visual-review-*.md`
2. 统计差距热力图（Layer 2.3）
3. 识别高频差距 Top 3
4. 对每个高频差距：
   - 已有对应策略 → 检查策略是否有效（验证次数够不够）
   - 无对应策略 → 新建策略
   - 策略验证 ≥ 5 次 → 考虑升级到 Layer 3

### 4.3 Prompt 策略的生命周期

```
新建（实验中）
  ↓ 被使用 ≥ 3 次
已验证
  ↓ 被使用 ≥ 5 次 且 效果稳定
固化候选
  ↓ 审查通过
固化到 SKILL.md（变成规则）
  ↓ 或者在实战中被突破条件频繁触发（>50%）
降级回策略（从 SKILL.md 移回策略库）
```

## 5. Layer 3: 规则固化（低频大更新）

### 5.1 触发条件

**以下任一条件满足时，考虑更新 SKILL.md：**

1. 一条 Prompt 策略在 Layer 2 验证 ≥ 5 次且效果稳定
2. 差距热力图显示某类问题持续恶化（>10 次未改善）
3. 发现新的 AI 痕迹模式（出现 ≥ 3 次且不在 Section 7 中）
4. 子 Agent 屡次违反某条规则（说明规则描述不够清晰）

### 5.2 固化流程

1. 写一个具体的规则条目（SKILL.md Section 7 格式：编号/痕迹/原因/修复/突破条件）
2. 检查与现有规则是否冲突
3. 如果是新增规则 → 追加到对应子节
4. 如果是修改现有规则 → 更新突破条件或修复方法
5. 如果是降级规则 → 从 SKILL.md 移回策略库，标注降级原因

### 5.3 降级流程（反向固化）

SKILL.md 中的规则如果：
- 子 Agent 在实战中 >50% 的时间触发了突破条件
- 或者规则本身导致了更差的结果

→ 从 SKILL.md 移除，放回策略库，标注：
```
状态: 降级自 SKILL.md
降级原因: 实战中 [场景] 下频繁不适用
原编号: T01
降级日期: YYYY-MM-DD
```

**规则不是永远正确的。** 进化引擎的核心能力之一是让过时规则退场。

## 6. 归因模型

### 6.1 五大归因类别

| 类别 | 含义 | 典型信号 |
|------|------|---------|
| **品味判断** | 旋钮值不对、美学方向偏离 | "不够高端""太花""不像品牌" |
| **Prompt 策略** | 给生图模型/AI 的指令不够精确 | 生图输出总是差一点、子 Agent 产出偏离预期 |
| **技术执行** | 代码/CSS/HTML 实现有问题 | 响应式坏掉、动效卡顿、排版错位 |
| **素材质量** | 图片/图标/占位图选择不当 | 占位图风格不匹配、图标不够专业 |
| **用户沟通** | 需求理解偏差 | "不是我要的""方向搞错了" |

### 6.2 归因 → 行动映射

| 归因 | Layer 2 行动 | Layer 3 行动（如果持续发生） |
|------|-------------|--------------------------|
| 品味判断 | 优化 Design Read 流程，调整旋钮预设 | 更新 Section 1 旋钮预设表 |
| Prompt 策略 | 新建/优化 Prompt 策略 | 固化策略为 SKILL.md 规则 |
| 技术执行 | 记录技术笔记，补充到项目文档 | 更新 SKILL.md Section 7.F 代码痕迹 |
| 素材质量 | 优化生图 prompt / 资产优先级链 | 更新 SKILL.md Section 6 |
| 用户沟通 | 优化 Section 0 信号读取流程 | 更新 SKILL.md Section 0.A |

### 6.3 归因不准时

如果差距无法清晰归入任何一个类别：
- 标记为「归因待定」
- 在下次类似任务时重点观察
- 累积 ≥ 3 次仍无法归因 → 提升为独立问题，人工介入分析

## 7. 进化节奏

### 7.1 每次任务

- Layer 1 强制复盘（除微调/修复外）
- 新策略实验 → 记录验证结果

### 7.2 每周

- Layer 2 批量蒸馏（扫描本周所有复盘）
- 更新差距热力图
- 清理已验证策略的状态标记

### 7.3 每月

- Layer 3 规则固化审查
- 检查是否有规则需要降级
- 更新 SKILL.md（如果需要）
- 回顾策略库，合并重复策略

### 7.4 每季度

- 全面审查 SKILL.md + 策略库 + 热力图
- 评估视觉输出能力的整体趋势（改善/持平/恶化）
- 调整进化引擎本身的参数（如：验证次数阈值、复盘频率）

## 8. 与现有体系的关系

| 现有体系 | 进化引擎的关系 |
|---------|--------------|
| SKILL.md Section 7 痕迹目录 | Layer 3 的产物。痕迹目录中的每条规则都应该有来自 Layer 2 的验证记录 |
| SKILL.md Section 12 循环进化 | 进化引擎的具体化实现。Section 12 是一句话描述，本文件是完整协议 |
| shared-knowledge/ | Layer 2 策略库 + 差距热力图的存储位置 |
| 项目 .openclaw-project/ | Layer 1 复盘记录的存储位置 |
| 三省六部制 | 进化引擎是独立的元能力，不依赖三省六部制。但如果视觉任务走三省六部制，Layer 1 复盘应嵌入交付流程 |
